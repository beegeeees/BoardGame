# 불편마불 합본 통합 메모

이 문서는 현재 `BulpyeonMabul` 앱 합본을 `BoardGame` 서버/로비 코드와 다시 합칠 때 확인해야 할 내용을 정리한 것이다.

## 현재 앱 합본 기준

- 메인 보드: `com.example.bulpyeonmabul.MainActivity`
- 점수 로직: `com.example.bulpyeonmabul.score.*`
- 공통 결과 규격: `com.example.bulpyeonmabul.GameContract`
- 서버 연결용 임시 경계면: `com.example.bulpyeonmabul.integration.MainGameServerBridge`
- 로컬 테스트용 더미 구현: `LocalTestServerBridge`

현재 앱은 서버 없이도 테스트할 수 있도록 4인 로컬 진행 흐름을 유지한다. 실제 멀티 완성 시에는 `LocalTestServerBridge` 대신 서버/로비 쪽 구현체를 연결하면 된다.

## 보드 진행 순서

룰_초안2 기준으로 다음 순서를 구현했다.

1. 1라운드: 4명의 플레이어가 각각 1턴 진행
2. 1라운드 종료 후 캡챠 미니게임
3. 2라운드: 4명의 플레이어가 각각 1턴 진행
4. 2라운드 종료 후 비밀번호 미니게임
5. 3라운드: 4명의 플레이어가 각각 1턴 진행
6. 3라운드 종료 후 볼륨 미로 미니게임
7. 누적 점수 기준 최종 순위 표시

현재 로컬 테스트에서는 사용자가 1P 미니게임만 직접 플레이하고, 나머지 플레이어의 미니게임 진행도는 임시 랜덤값으로 채운다. 멀티 연동 시 이 부분은 서버가 받은 각 클라이언트 결과로 대체해야 한다.

## 미니게임 결과 Intent 규격

모든 미니게임은 Activity 결과로 아래 값을 반환하도록 맞췄다.

- `GameContract.EXTRA_SUCCESS`: 성공 여부 `boolean`
- `GameContract.EXTRA_PROGRESS`: 진행도 `int`, 0~100
- `GameContract.EXTRA_DICE_RESULT`: 주사위 결과 `int`, 1~6
- `GameContract.EXTRA_ROUND`: 라운드 번호 `int`
- `GameContract.EXTRA_PLAYER_INDEX`: 로컬 플레이어 인덱스 `int`

## 주사위 규칙 반영 상태

`주사위_규칙_수정.docx` 기준으로 맞췄다.

- 버튼: `1~2`, `3~4`, `5~6`, `랜덤`
- 확률:
  - `1~2`: 30%, 30%, 10%, 10%, 10%, 10%
  - `3~4`: 15%, 15%, 20%, 20%, 15%, 15%
  - `5~6`: 15%, 15%, 15%, 15%, 20%, 20%
  - `랜덤`: 전부 1/6
- 제한시간 10초
- 버튼 미선택 시간초과 시 랜덤 주사위
- 버튼 선택 후 시간초과 시 선택된 주사위
- 버튼 선택 때마다 20% 확률로 야바위 다크 패턴
- 야바위 후 타이머 리셋 및 선택 즉시 자동 굴림

현재 `DiceActivity`는 결과를 `EXTRA_DICE_RESULT`로 반환한다.

## 광고 칸 마이크로게임 규칙 반영 상태

`마이크로게임 규칙.docx` 기준으로 광고 칸에서는 캡챠가 뜨지 않고 아래 3개 중 하나가 랜덤 실행된다.

- `AdGame1Activity`: 진짜 X 버튼 찾기
- `AdGame2Activity`: 정확한 시간 맞추기
- `AdGame3Activity`: 빠른 연타로 게이지 채우기

공통 점수:

- 성공: +5점
- 실패 또는 시간초과: -5점

서버에는 실시간 터치/타이머 값을 보내지 않고, 종료 시점의 성공/실패만 보내면 된다.

## 라운드 미니게임

- 1라운드: `CaptchaActivity`
- 2라운드: `PasswordActivity`
- 3라운드: `VolumeMazeActivity`

주의: `title` 프로젝트의 볼륨 미로 원본은 Kotlin 파일이었다. 현재 앱 합본에서는 “Kotlin 파일은 Java로 변경” 요청에 맞추기 위해 Java 기반 `VolumeMazeActivity`/`VolumeMazeView`로 이식했다. 자석이 외곽을 돌고 공이 자석에 끌려가며 미로 벽에 충돌하는 원본 구조를 유지하도록 옮겼다.

## BoardGame 서버 코드와 맞물리지 않는 점

현재 `BoardGame` 서버 쪽 구현은 룰_초안2 및 현재 앱 합본과 차이가 있다. 서버는 팀원이 계속 수정할 예정이므로 이 앱에서는 서버 코드를 직접 고치지 않았다.

- 서버 `BoardGameService.rollDice()`는 서버가 랜덤 주사위를 직접 굴린다. 현재 앱은 클라이언트 주사위 컨트롤/야바위 결과를 확정한 뒤 서버로 보내야 한다.
- 서버 타일 배치는 `%8`, `%6`, `%5` 기반이다. 현재 앱 보드 배치는 16칸 고정 배치다.
- 서버 미니게임 점수는 `30/20/10/5`이고, 현재 앱 점수 엔진은 `10/7/5/3`이다. 룰_초안2에는 “추천 점수” 표가 비어 있어, 기존 title 점수 엔진에서 가져온 값을 유지했다.
- 서버는 마이크로게임 타입을 `QUICK_TAP`으로만 시작한다. 현재 앱은 광고 칸에서 3종 마이크로게임 중 하나를 랜덤 실행한다.
- 서버는 라운드별 미니게임을 자동으로 `캡챠 -> 비밀번호 -> 볼륨 미로` 순서로 고정하는 로직이 아직 앱 합본과 동일하다고 확인되지 않았다.
- 서버가 공식 점수를 관리할 경우, 앱의 `BoardScoreController`는 로컬 테스트/표시용으로 두고 최종 점수 반영은 서버 스냅샷으로 덮어야 한다.

## 서버 연결 시 권장 작업

1. 로비에서 받은 `roomCode`, `playerId`, `playerIndex`, `nickname`을 `MainActivity`에 전달한다.
2. `MainGameServerBridge` 구현체를 만들고 `MainActivity`의 `LocalTestServerBridge`를 교체한다.
3. 주사위 결과는 클라이언트가 확정한 값과 도착 위치를 서버로 보낸다.
4. 광고 마이크로게임은 종료 시 성공/실패만 서버로 보낸다.
5. 라운드 미니게임은 진행도 또는 성공/실패를 서버로 보낸다.
6. 서버가 4명 결과를 모아 순위를 계산한 뒤, 앱은 서버 순위를 받아 `BoardScoreController.onMiniGameFinished()` 또는 서버 스냅샷 UI 반영으로 처리한다.

## 현재 남겨둔 임시 기능

- 서버 없이 `MainActivity`에 바로 접속 가능하다.
- 모든 플레이어 점수판은 한 기기에서 표시된다.
- 1P 점수판에 `나` 표시가 붙는다.
- 현재 턴 플레이어 점수판은 테두리가 강조된다.
- 1P 외 미니게임 결과는 로컬 테스트를 위해 임시 랜덤 진행도로 처리한다.
- 메인 보드 하단의 `볼륨 미로 테스트` 버튼은 테스트용 임시 진입 버튼이다. 최종 합본에서는 제거해야 한다.

최종 멀티 완성본에서는 위 임시 로컬 테스트 흐름 중 1P 외 랜덤 결과 생성 부분을 제거해야 한다.
